# 画面項目定義書

## 0. 目的・対象画面・参照ドキュメント

### 目的
確定済みの正規化データモデル（`households` / `household_members` / `deceased_persons` / `districts` / `cemetery_plots` / `columbarium_units`）を前提に、檀家管理・過去帳の画面に表示する項目を確定する。全表示項目について、どのテーブル・カラム（または導出式）に由来するかを明記し、現行 UI（モックデータ実装）からの差分を洗い出すことで、以後の API 設計・フロント実装の入力とする。

### 対象画面（4画面 + 編集モーダル2種）
1. 檀家管理 — 一覧
2. 檀家管理 — 詳細
3. 過去帳 — 一覧
4. 過去帳 — 詳細
5. （付随）檀家 新規登録・編集モーダル
6. （付随）過去帳 新規登録・編集モーダル

### 参照ドキュメント
- [データモデル設計.md](../DB設計/データモデル設計.md) — エンティティ構成・ER図・enum定義・設計判断の根拠
- [テーブル定義書.md](../DB設計/テーブル定義書.md) — 物理カラム定義（型・桁数・NULL許可・制約）
- [データ移行設計.md](../DB設計/データ移行設計.md) — 列単位の移行マッピング・クレンジング課題・要ヒアリング事項
- [檀家一覧_項目整合表.md](檀家一覧_項目整合表.md) — 檀家一覧の表示列と実装コード（型・API・SQL・DBカラム）の整合表（実装後トレーサビリティ）

### 前提とする方針（`.claude/feature.md` 準拠）
- 檀家一覧から家紋・宗派・檀家番号を削除。過去帳一覧にヘッダを実装。
- 檀家詳細・過去帳詳細ともに「編集する」ボタン押下でモーダル編集を開く。削除は檀家編集モーダル内に移設（檀家詳細本体には置かない）。
- 案内送信機能・LINE関連機能は対象外（削除済み）。

| 検索用項目 | 由来 | 備考 |
| --- | --- | --- |
| フリガナ | `household_members.name_kana`（`member_role='head'`） | 一覧列は持たず、検索ボックスの部分一致対象にのみ含める |
| 住所 | `households.address_1` | 同上（`address_2` は移行メモ流用が残るため検索対象外） |

### 1.2 既定ソート・フィルタ・SP非表示列

- **既定ソート**: 家名（50音順）昇順。
  - 理由: 現行既定の「最近のお参り」はお参り記録機能の削除に伴い項目自体を廃止したため、代替の既定ソートキーが必要。家名は主表示・検索キーであり最も直感的なため既定とする。
- **ソート選択肢**: 家名 / 地区 / 過去帳人数（多い順）。
- **フィルタ**: 地区（`households.district_id` の区分1→区分2 二段階選択）、関係区分（`households.relation_type`）。
  - 宗派フィルタは削除済み（`.claude/feature.md`）のため、この 2 項目を代替として提案する。
- **モバイル（SP）時に非表示にする列**: 戸主名・地区・関係区分・電話・過去帳人数・直近の予定（家名のみ表示し、行タップで檀家詳細に遷移して全項目を確認する）。
  - 根拠: 現行実装（`app/frontend/src/styles/sp.css` の `@media (max-width: 640px)` ブロック）で `.ptbl-row` は `.c-name` 以外の列（`.c-head` `.c-mem` `.c-visit` `.c-anc` `.c-sched`）を `display: none` にしている。新項目構成でも同じレスポンシブ方針（1行=家名のみ、詳細はタップで確認）を踏襲する。
- **ページネーション**: サーバーサイドページネーション（1ページ20件）。テーブル下部に「前へ／次へ」ボタンとページ選択セレクトを表示する。検索・フィルタ・ソートのいずれかを変更した場合は1ページ目に戻る。

### 1.3 現行UIとの差分

| 区分 | 項目 | 理由 |
| --- | --- | --- |
| 削除 | 家紋 | `.claude/feature.md` 明記の削除方針。データモデル上も `households`/`household_members` に家紋カラムなし |
| 削除 | 宗派 | 同上。データモデルに宗派カラムが存在しない（寺院は単一宗派前提） |
| 削除 | 檀家番号 | 同上。旧家番・旧整理番号は本番スキーマに持たず、ステージングテーブル `import_records` の `raw_data` にのみ保持されるため業務表示対象外（末尾「表示しないカラム」表参照） |
| 削除 | 家族数（現行 `members`） | 承認プランの削除対象。家族構成は詳細画面で `household_members` を実データ表示するため、一覧の概数表示は不要と判断 |
| 削除 | 最近のお参り（現行 `lastVisit`） | お参り記録機能自体を削除（`.claude/feature.md`）。データ源も消滅 |
| 削除 | 入檀年（現行 `joined`） | 承認プランの削除対象。移行元データに対応カラムなし（`households` に加入年カラムなし） |
| 追加 | 地区 | `districts` が正規化マスタとして新設されたため、業務上重要な分類軸として一覧に追加 |
| 追加 | 関係区分 | `households.relation_type`（檀家/信徒/檀徒等）は実データに存在する重要区分。宗派削除の代替でもある |
| 追加 | 電話 | `households.phone` は実データに存在し、一覧からの一次連絡用途で有用なため追加 |
| 変更 | 過去帳（現行 `ancestors` 件数） → 過去帳人数 | 表示概念は同じだが、由来を `deceased_persons` の COUNT 導出として明確化 |
| 維持（将来接続） | 直近の予定 | 現行はモック文字列（`scheduled`）。将来のスケジュール機能とのデータ連携を前提に項目自体は維持 |

`.claude/feature.md` の方針により、檀家詳細の「編集する」ボタン押下でこのモーダルを開く（新規登録ダイアログと共用）。フォーム項目は `households` の編集可能カラム、および 1 世帯 1 件の現戸主（`household_members`, `member_role='head'`）の氏名・フリガナに限定する。

| 項目 | 対応カラム | 入力形式 | 必須 | 新規時初期値 | 備考 |
| --- | --- | --- | --- | --- | --- |
| 家名 | `households.family_name` | テキスト | ○ | 空 | 檀家一覧の主キー表示。全角スペース等は保存時にトリム |
| 戸主名 | `household_members.name`（`member_role='head'`） | テキスト | ○ | 空 | 新規時は head 行を新規作成。既存編集時は該当 head 行を更新 |
| フリガナ（戸主） | `household_members.name_kana`（`member_role='head'`） | テキスト | − | 空 | 全角カナ入力を推奨する説明文を添える |

| 関係区分 | `households.relation_type` | セレクト（檀家/信徒/檀徒/他寺/その他） | − | 未選択 | |
| 状態 | `households.status` | セレクト（active/inactive/deleted） | − | `active`（新規時は固定・非表示） | 編集時のみ表示。`deleted` へ変更する操作は削除確認ダイアログ経由に限定し、本フィールドから直接選ばせない（1.4 節「状態」欄の運用方針を参照） |
| 郵便番号 | `households.postal_code` | テキスト（ハイフンあり） | − | 空 | |
| 住所1 | `households.address_1` | テキスト | − | 空 | |
| 住所2 | `households.address_2` | テキスト | − | 空 | |
| 電話 | `households.phone` | テキスト | − | 空 | |

| 施食 | `households.sejiki_service` | セレクト（組/郵送/なし/不要） | − | 未選択 | |

### 3.2 状態（status）の表示ポリシー

- **active**: 通常表示。バッジなし。全操作（編集・過去帳閲覧）を通常どおり許可。
- **inactive**（離檀等）: 檀家一覧では既定非表示とし、一覧フィルタで「離檀世帯を含む」を選んだ場合のみ表示する。詳細画面ではヘッダ直下に「離檀」バッジ（グレー）を表示し、住所2 に残る離檀メモ原文がある場合はそのまま備考欄付近に表示する。編集・過去帳閲覧は引き続き許可する（記録保持の目的のため）。
- **deleted**（soft delete）: 檀家一覧には表示しない。詳細画面への直接遷移（URL直打ち等）があった場合のみ「削除済み」バッジ（赤）と警告バナー「この世帯は削除済みです」を表示し、編集ボタンは非活性化する（過去帳リスト・区画情報は閲覧のみ可）。復元導線は本設計のスコープ外とし、必要になった場合は別途設計する。

| 区分 | 項目 | 理由 |
| --- | --- | --- |
| 削除 | 家紋 | 一覧同様、データモデルにカラムなし |
| 削除 | 宗派 | 同上 |
| 削除 | 加入年（「加入 1972年 (54年)」表示） | データモデルに加入年カラムなし。移行元にも対応列がない |
| 削除 | 最近のお参り | お参り記録機能の削除に伴い項目自体を廃止 |
| 削除 | 案内送信機能 | `.claude/feature.md` 明記の削除方針 |
| 削除 | 檀家削除ボタン（詳細画面本体） | `.claude/feature.md` の方針により編集モーダル内へ移設 |
| 追加 | フリガナ・〒住所・携帯 | 実データ（`household_members.name_kana` / `households.postal_code` / `mobile_phone`）に存在するが現行 UI 未表示のため追加 |

| 変更 | 「過去帳を見る」導線 → この世帯の過去帳リスト | 現行はボタン1つで過去帳画面へ遷移するのみ。実データでは世帯に紐づく故人が複数存在しうるため、詳細内にリスト表示し各行から遷移できるようにする |

| 没年月日（西暦+和暦併記） | `deceased_persons.death_year` / `death_month` / `death_day` / `death_date` / `death_wareki_raw` | 「1998年5月16日（平成10年5月16日）」等。不完全日付は 4.2 節の規則に従う | − | ○ | |
| 享年 | `deceased_persons.age_at_death` | 数値 + 「歳」 | − | − | NULL は「不詳」 |
| 次の年忌 | 導出（4.3節の式） | チップ（例: 「七回忌 あと38日」「三十三回忌 済」） | − | ○（既定） | |

検索対象を満たすための補足: 戒名・俗名は完全一致に加え部分一致、家名は `households.family_name` の部分一致（`household_id` 経由の JOIN）。

### 4.2 不完全日付の表示規則（提案）

過去帳の没年月日は `death_year` / `death_month` / `death_day` を個別カラムで保持し、3つ揃った場合のみ `death_date` を導出格納する設計（[データモデル設計.md](../DB設計/データモデル設計.md) 4.4節）。画面表示では西暦を主とし、和暦（`death_wareki_raw`）を括弧併記する。以下のケース分けで表示する。

| ケース | 条件 | 一覧・詳細共通の表示例 | 補足 |
| --- | --- | --- | --- |
| 完全 | `death_year`・`death_month`・`death_day` すべて非NULL（`death_date` あり） | `1998年5月16日（平成10年5月16日）` | 和暦は `death_wareki_raw` をそのまま括弧内に表示 |
| 年月のみ | `death_year`・`death_month` 非NULL、`death_day` NULL | `1998年5月（日不詳）（平成10年5月）` | |
| 年のみ | `death_year` 非NULL、`death_month`・`death_day` ともNULL | `1998年（月日不詳）` | 設問で例示された表記をそのまま採用 |
| 年月日すべて不明 | `death_year` も NULL | `不詳` | 過去帳の中でも極めて稀（家番8888等の一部を想定） |
| 和暦欠落 | 上記いずれかの西暦表示は可能だが `death_wareki_raw` が NULL | 括弧部を省略し西暦のみ表示（例: `1998年5月16日`） | |
| 和暦が元号省略表記（例「十年」） | `death_wareki_raw` が数字のみ等、元号を含まない原文 | 西暦から機械算出した元号を優先表示し、原文は詳細画面の戒名表記注意欄付近に「※原表記: 十年」として併記 | 一覧では算出済みの元号表示のみ（例: `1998年5月16日（平成10年5月16日）`）とし、原文注記は詳細画面のみに出す |

### 4.3 「次の年忌」の導出式

年忌は保持せず、没年月日と基準年（システム日付が属する年）から機械的に算出する（[データモデル設計.md](../DB設計/データモデル設計.md) 4.4節の設計判断に基づく）。マイルストーンは**実データの年忌グループ（12種）に合わせる**。過去帳 CSV の年忌列には 三十七回忌・四十三回忌・四十七回忌 が存在し（グループ見出しの遷移: 一周忌→三回忌→七回忌→十三→十七→二十三→二十七→三十三→三十七→四十三→四十七→五十回忌）、この寺は 12 種の年忌を実施しているのが実態である。現行実装 `app/frontend/src/features/memorial/utils.ts` の年忌テーブルは 9 種（三十七・四十三・四十七回忌を含まない）のため、**UI 実装時にテーブルの拡張が必要**。

1. 年忌マイルストーン一覧（回忌名, 没年からの経過年数 n・全12種）: 一周忌(1) / 三回忌(2) / 七回忌(6) / 十三回忌(12) / 十七回忌(16) / 二十三回忌(22) / 二十七回忌(26) / 三十三回忌(32) / 三十七回忌(36) / 四十三回忌(42) / 四十七回忌(46) / 五十回忌(49)
2. 各マイルストーンの実施年 = `death_year + n`
3. 実施月日は `death_month` / `death_day` が完全な場合はそれをそのまま使用（没日と同月同日に実施する寺務慣行のため）。`death_month` または `death_day` が NULL の場合、実施月日も確定できないため「次の年忌: ◯◯回忌（実施年度: 20XX年度、月日未定）」と表示する。
4. 「次の年忌」＝ 実施年月日が基準日（システムの「今日」）以降で最も近いマイルストーン。該当なし（すべて経過済み）の場合は最終マイルストーン（五十回忌）に「済」バッジを付けて表示する。

### 4.4 既定ソート・フィルタ・SP非表示列

- **既定ソート**: 次の年忌が近い順（現行実装の `upcoming` 相当を踏襲）。
- **ソート選択肢**: 次の年忌が近い順 / 没年月日が新しい順 / 家名順。
- **フィルタ**: 家（`households.family_name`、`household_id` 経由のセレクト）。宗派フィルタは寺院が単一宗派前提のためデータモデルに存在せず対象外。
- **モバイル（SP）時に非表示にする列**: 没年月日・享年（戒名・俗名・家名・続柄・次の年忌チップのみを1行に表示し、行タップで過去帳詳細に遷移して確認する）。
  - 根拠: 現行実装（`sp.css` `@media (max-width: 640px)` 内 `.memorial-row .mr-date { display: none !important; }`）で没年月日列（享年を含む）を非表示にしている。新項目構成でも同じ方針を踏襲する。
- **ページネーション**: 1ページ20件（ページャ UI は檀家一覧と共通の `Pagination` コンポーネント）。ただし過去帳は「年忌が近い順」ソートと統計カードをフロント側の全件算出（`nextAnniversary` 導出）に依存するため、サーバーサイドではなく**クライアントサイドページング**（全件取得後に表示のみ分割）とする。検索・家フィルタ・並び順の変更時は1ページ目に戻る。
