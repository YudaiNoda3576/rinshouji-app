# テーブル定義書

## 拡張方針サマリ（人中心モデルへの移行）

### 既存設計からの主な差分

- `persons`（個人マスタ）を新設し、檀家世帯の代表者・家族・前戸主・連絡先人物、檀家ではない参拝者・ボランティア・寄進者・業者・僧侶・寺族などを同じ個人マスタで扱う。
- `households` は檀信徒名簿ベースの世帯管理として維持する。家番 `house_no`、代表者氏名原表記、住所、地区、関係区分は後方互換のため残し、個人マスタへ寄せられた代表者は `representative_person_id` で参照する。
- 旧 `household_related_people` 相当の永続データは `household_person_relationships` に統合する。氏名だけの関係者も移行時に `persons` を作成し、世帯との関係・元列名・原表記を関連テーブルに保持する。
- 檀家、参拝者、ボランティア、寄進者、業者、僧侶、寺族などの「寺院との関係」は `person_roles.stakeholder_code` で表現する。役割は固定カラムにせず、複数付与と期間管理を可能にする。
- ログイン・権限は `user_accounts.role` の単一ロールで管理する。ステークホルダー種別は「人の社会的・業務的な関係」、アカウントロールは「アプリ操作権限」として分離する。
- 年忌記録 `deceased_memorial_anniversaries` は「故人ごとの年忌対象年」を管理するテーブルとして完結させる。年忌種別、対象年、原表記を保持し、年忌対象年の抽出・確認に利用する。
- 人中心モデル、役割管理、認証、移行監査は将来拡張の土台として維持する。固定値の小マスタと本格RBACは、YAGNIに基づき当面はenum/CHECK制約または単一カラムで持ち、必要時にテーブル展開する。

### バランス整理の設計判断

- 少数で固定的な区分・状態はテーブル化せず、参照元テーブルの `*_code` カラムに集約する。コードは英字の安定値、表示名はアプリケーション定義または近接の区分値一覧で管理する。
- 本格RBACは現スコープでは過剰なため廃止し、`user_accounts.role` で単一ロールを持つ。部署別・操作別・期間付き権限が必要になった段階で、ロール・権限・関連テーブルへ分離する。
- 施設区画は元データが自由記述で汚く、正規化済み区画マスタを先に作ると移行判断が重くなる。当面は `household_facility_usages` に施設種別コード・状態コード・区画原表記・幅・入金・代金・備考をまとめて保持し、データ清掃後に区画マスタ化を検討する。
- `household_memorial_tablets` は位牌が墓地・納骨堂区画と性質が異なるため、無理に施設利用へ統合せず単独維持する。
- `memorial_anniversary_types` は年忌名だけでなく `years_after_death`、表示順、有効フラグを持ち、年忌案内ロジックで使うためマスタとして維持する。

### 移行方針

1. 既存CSVは引き続き `import_batches`、`source_csv_rows` に全行保持する。
2. `households.representative_name` から代表者候補の `persons` を作成し、確度が高いものだけ `households.representative_person_id` と `household_person_relationships.is_representative = true` に接続する。
3. 旧 `household_related_people` に相当する `家族*`、`前戸主*`、`納骨堂位牌名` などは、個人として扱うべきものを `persons` 化し、世帯との関係は `household_person_relationships` へ移す。位牌名など個人同定に不向きな値は原表記を保持し、必要に応じて `household_memorial_tablets` に残す。
4. `households.relationship_code` は檀家・信徒などの世帯単位区分を表す。`person_roles.stakeholder_code` は個人単位のステークホルダー区分を表す。世帯が檀家であることと、世帯内の誰が本人アカウントを持つかは分けて管理する。
5. 移行直後に旧テーブル名を参照する画面・帳票が残る場合は、`household_related_people` を互換ビューとして提供する。新規実装では `persons` と `household_person_relationships` を参照する。

### 個人情報・認証情報の扱い

- 生パスワードは保持しない。`user_accounts.password_hash` にはArgon2id、bcrypt等の適応的ハッシュを保存し、ハッシュ方式・パラメータを `password_hash_algorithm` に記録する。
- OAuth等の外部認証IDは `user_external_identities` に分離し、プロバイダ内のsubjectを一意キーとして扱う。プロバイダ由来メールはログインメールと一致しない場合があるため別カラムに保持する。
- `source_csv_rows.raw_record`、`persons`、`person_contact_points`、`user_accounts` は個人情報・認証情報を含む。`user_accounts.role` に基づき、寺務管理者など必要最小限のロールだけにアクセスを許可する。
- 退会・表示停止・誤登録の取り扱いのため、個人・アカウント・世帯関連などの業務データには `deleted_at` を持たせる。法的・会計的に保存が必要な記録は物理削除せず、表示抑止または匿名化方針を別途定める。
- 監査用途で原CSVを残す一方、画面では原行JSONを直接表示しない。検索・帳票で必要な項目は正規化カラムを経由する。

## 共通方針

- PostgreSQLを前提とする。メールアドレスの大文字小文字差異を吸収するため、`citext` 拡張の利用を想定する。
- 物理名は `snake_case`、論理名は日本語とする。
- 全テーブルに `created_at`、`updated_at` を持たせる。`updated_at` は更新トリガーで現在時刻に更新する。
- 家番は `char(4)` とし、`^[0-9]{4}$` を満たす値だけを正規化済み家番として扱う。
- CSVの原表記は、業務上の正規化カラムとは別に `raw_*` カラム、または `source_csv_rows.raw_record` に保持する。
- 固定値の区分・状態は、参照元テーブルのコード列とCHECK制約で管理する。コードは安定識別子として使い、表示名は区分値一覧に定義する。
- コード値が業務上増減し、属性・並び順・有効期限・権限制御を独自に持つ段階になったら、該当コード列をマスタテーブルへ展開する。

## データ品質メモ / 移行時の注意点

### 檀信徒名簿

- カラムは33列。最後の列はヘッダー空欄だが、795行に `0`、`1`、`2`、`3` が入っているため破棄せず `households.source_sort_rank` に保持する。
- データ行1,175行のうち、完全空行は3行。非空行は1,172行。
- `家番` 非空は1,011行、4桁家番のユニークは999件。重複家番は9グループ/21行あり、`0749` は5行、`0141`、`0395`、`0553`、`0797`、`0800`、`0818`、`0821`、`8101` は各2行。移行時に代表行選定が必要。
- 家番空欄だが内容がある行が161行ある。墓地・納骨堂の空き区画、合葬墓、他檀徒、メモ行が混在するため、世帯としては取り込まず `source_csv_rows` に保持し、世帯利用として確定できるものだけ `household_facility_usages` へ移す。
- `地区区分１` は地区だけでなく、`削除`、`墓地`、`納骨堂`、`墓地空き`、`納骨堂空き`、`墓地堂空き`、`墓地、納骨堂` など管理用カテゴリも混在する。
- `関係` は `檀家` 601件、`信徒` 120件、`檀徒` 102件、`他寺` 54件、`その他` 18件、`壇徒` 1件。`壇徒` は表記揺れとして `relationship_code = 'danto'` に寄せる。
- `氏名` と `ﾌﾘｶﾞﾅ` から `persons` を作る場合、同姓同名や世帯代表者交代がありうるため自動同定を過信しない。移行時は `households.representative_name` も残す。
- `家族１`、`前戸主１` などは `persons` と `household_person_relationships` へ移す。ただし位牌名・屋号・メモのように個人同定できない値は、無理に `persons` 化せず原表記を残す。
- `位牌区分` は `あり` 641件、`なし` 344件のほか、`？？`、`なし「` がある。`tablet_status_code` へ寄せつつ原表記を保持する。
- `棚経` は `８月`、`８月１４`、`７月１３` のような日程値、`郵送`、`なし`、`不要`、`月経で` が混在する。`service_status_code` と日程を分離して移行する。
- `月経` は日付数値が主だが、`第２木`、`７,13`、`３9,26`、半角数字が混在する。正規化できないものは `schedule_note` に保持する。
- `納骨堂区分`、`墓地区分` は `なし` や `返還` の状態値と、`中北北ろ５ー４`、`東北１３Ｃ` のような区画値、自由記述が混在する。`facility_status_code`、`location_raw`、`raw_section_value` に分けて移行する。
- `墓地幅` は543件中541件が数値化可能。`290⇒140`、`75+30` は `width_raw` に保持し、確認後に `width_cm` を補正する。
- `墓地入金` は `42, 6,13` のような昭和年らしき略記、`H3, 5,` のような和暦略記がある。`payment_date_raw` を必ず保持する。
- 郵便番号に全角数字混入、電話番号に `???`、`***`、携帯電話に人物名付き値があるため、原表記と正規化値を分ける。本人連絡先として確定できたものだけ `person_contact_points` にも登録する。

### 過去帳

- カラムは14列。最後の列はヘッダー空欄で、1行だけ値が入っているため `source_csv_rows.raw_record` に保持する。
- データ行2,387行のうち、完全空行は64行。年ごとの区切り行、年忌見出し行、家番空欄の故人行が混在する。
- `家番` 非空は2,195行、4桁家番は2,138行。4桁家番のうち檀信徒名簿に存在しないものは `8888` 57行、`0000` 3行、`0803` 1行。
- 4桁以外の家番は57行あり、`削除` 9行、`離檀` 5行、`8888*` 4行、その他 `8014*` などアスタリスク付き家番がある。アスタリスク付きは元値を `raw_house_no` に残し、正規化先家番を個別確認する。
- `年忌` は442件。通常値は `一周忌`、`三回忌`、`七回忌`、`十三回忌`、`十七回忌`、`二十三回忌`、`二十七回忌`、`三十三回忌`、`三十七回忌`、`四十三回忌`、`四十七回忌`、`五十回忌`。ただし `令和5年1月15日27回忌済`、`安藤信実` のような混入値がある。
- `deceased_memorial_anniversaries` は年忌対象年の記録であり、年忌種別・対象年・原表記をこのテーブルで管理する。
- `西暦` は主に1960年から2025年だが、末尾付近に西暦空欄で `大正十五`、`昭和十`、`昭和２４年` など邦暦のみの行がある。
- `月` は漢数字が主だが、`10`、`十一　十七` がある。`日` は漢数字が主だが、`3`、`乙酉`、`丙午`、`乙巳（木の弟）` がある。
- `享年` は数値が主だが、`死産`、`７ヶ月`、`36時間`、`1年８ヶ月`、`当才`、`？` などがある。数値化できないものは `age_at_death_raw` に保持する。
- 過去帳の `地区１/２` は現行名簿の `地区区分１/２` と一致しない行が多い。歴史的スナップショットとして `deceased_persons.district_raw_1/2` を保持し、世帯の現地区は更新しない。

## 移行順序

1. `import_batches`、`source_csv_rows` に2つのCSVを全行取り込む。
2. `districts` と `memorial_anniversary_types` を作成する。その他の固定区分はテーブルを作らず、各コード列のCHECK制約に従ってマッピングする。
3. 檀信徒名簿の4桁家番から `households` を作成する。重複家番は代表行を決め、候補行を監査対象に残す。
4. 世帯代表者・家族・前戸主などから `persons` を作成し、`household_person_relationships` へ紐付ける。確度が低い人物候補は原表記を残して手動確認に回す。
5. 電話番号、法要設定、施設利用、位牌情報を世帯へ紐付ける。個人本人の連絡先として確定できるものは `person_contact_points` にも登録する。
6. 世帯の `relationship_code` と世帯人物関連から、必要に応じて `person_roles.stakeholder_code = 'parishioner'` などのステークホルダー区分を付与する。
7. 過去帳の家番を正規化し、存在しない4桁家番は仮世帯作成または業務確認を行う。
8. 正規化できた過去帳行を `deceased_persons` に投入し、年忌値がある行は `deceased_memorial_anniversaries` に投入する。
9. 本人ログインを開始する段階で `user_accounts` を作成し、`role` に最小限のアプリロールを設定する。初期パスワードを平文保存せず、招待メールまたは一時トークン方式で初期設定させる。
10. OAuth等を利用する場合は `user_external_identities.provider_code` と `provider_subject` を登録する。
11. 家番空欄、特殊家番、区切り行、日付・享年の正規化不能値を一覧化し、業務確認後に補正する。
