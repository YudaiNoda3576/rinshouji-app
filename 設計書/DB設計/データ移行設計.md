# データ移行設計

既存 CSV（名簿・過去帳）から正規化モデル（[データモデル設計.md](./データモデル設計.md) / [テーブル定義書.md](./テーブル定義書.md)）への移行方針。全列マッピング・クレンジング課題・要ヒアリング事項・突合ウォークスルーを定義する。

## 1. 移行手順（2 段構成）

移行は **ステージング分離方式**（[データモデル設計.md](./データモデル設計.md) 「6.4」）に基づき、次の 2 段で行う。

| 段階 | 処理 | 内容 |
| --- | --- | --- |
| **①** | CSV → `import_records` 取込 | CSV 1 行 = 1 レコードとして、**全列（無名列含む）を `raw_data` JSONB に列名キーで丸ごと格納** する。`source_file`（檀信徒名簿/過去帳）+ `row_number` で一意化。クレンジング・変換は一切行わない（原文の正はここに一元化される）。過去帳の空スペーサ行 64 行はこの段階でスキップする |
| **②** | `import_records` → 本番変換 | 「3. 全列マッピング表」の規則に従い、raw_data から本番テーブル（districts / households / household_members / deceased_persons / cemetery_plots / columbarium_units）へ変換・生成する。生成した `households` / `deceased_persons` の各行には取込元の `import_record_id` を設定する |

- ①は追記専用・再実行可能（`UNIQUE(source_file, row_number)` で重複防止）。②は①の完了後に冪等に再実行できるよう、`import_record_id` を突合キーとして洗い替える。
- 移行検証は「② の結果 ↔ ① の raw_data」を `import_record_id` で突合して行う。本節以降のマッピング表は **② の変換規則** を定義する。
- パーティモデル導入（[パーティモデル設計.md](./パーティモデル設計.md)）に伴い、②の完了後に **③ パーティ生成**（`household_members` → `parties`/`household_memberships`/`party_roles` の機械生成）を実行する。③は②とは独立した節「7. フェーズ③: パーティ生成」に定義する。①②のマッピング・クレンジング方針（本節以降）はパーティモデル導入による変更を受けない。

### 再検証で事前分析と食い違った点
1. **過去帳の列数**: 事前分析は「13 列（最終無名列含む）」だが、実 CSV は **14 フィールド**（13 名前付き列 + 末尾無名列）。本書は全 14 列をマッピングする（総計 33 + 14 = **47 列**）。
2. **過去帳の行数**: 物理 2,387 行のうち 64 行が全項目空のスペーサ行。有効行は 2,323（邦暦充足 2,322・西暦充足 2,317）。事前分析「2,322 行」は有効行数に一致。
3. **墓地入金は入金状況ではなく入金日**: 値は「42, 6,13」「H3, 5,」等の和暦日付。既存定義書の「入金状況」は誤り。DATE + 原文保持へ変更。
4. **位牌区分に表記ゆれ 2 件**: あり641・なし344 に加え「？？」1・「なし「」1。
5. **年忌の非空は 442**（有効年忌ラベル 440 + 混入「安藤信実」1 +「令和5年…27回忌済」1）。事前分析「441」と 1 差。
6. **月経に複雑値**: 数値 121 件に加え「第２木」「７,13」「３9,26」「１５,9」の 4 件。非空計 125 は一致。
7. **住所2 の「離檀」メモは 8 件**（「離檀」6・「令和4年2月離檀」等日付付き 2）。`households.status='inactive'` への振替対象。
8. **氏名の単一トークンは 100 件**（名のみ・スペースなしフルネーム・「岩田家」「墓標０２」等が混在）、氏名欠損 33 世帯。うち前戸主から家名補完可能なのは 1 世帯のみ。姓名非分割の判断根拠（[データモデル設計.md](./データモデル設計.md) 6.5）。

## 3. 全列マッピング表

分類凡例（第 2 段の変換規則。**全 47 列は分類に関わらず第 1 段で `import_records.raw_data` に原文保持される**）:
- **写像** = ほぼそのまま本番カラムへ格納
- **変換** = 整形・分割・型変換・行分離して本番カラムへ格納
- **導出** = 他項目から算出して本番カラムへ格納
- **ステージのみ** = 本番カラムを持たない。`import_records.raw_data` にのみ保持（理由必須。突合キー等での利用は備考に記載）

## 5. 要ヒアリング事項

> 以下 13 項目はクライアント（林昌寺）向け確認資料 [クライアント確認事項.md](../クライアント確認事項.md) に転記済み。回答はそちらに集約する。

1. **名簿 無名列（0/1/2/3）**: 意味・業務用途。確定するまで import_records.raw_data に保持（本番カラムなし）。
2. **月経の意味**: 月参りの「日」なのか回数なのか。複雑値（第２木・７,13）の解釈。
3. **墓地備考の「本数」**: 「２本」等が墓石本数か区画数か。区画テーブルの粒度に影響。

## 7. フェーズ③: パーティ生成

パーティモデル導入（[パーティモデル設計.md](./パーティモデル設計.md)）に伴い、フェーズ②（`import_records` → 本番 6 テーブル）の完了後に **フェーズ③: パーティ生成** を追加する。`household_members`（檀信徒人物の正）から `parties` / `household_memberships` / `party_roles` を機械生成する処理であり、`household_members` 自体の内容は変更しない（読み取り専用の派生生成）。

### 7.1 生成規則

1. **ガード**: `origin='import'` の `parties` に `user_accounts` が付与されている行を検出し、あれば report に警告を出力して当該 `party` を削除対象から除外する（運用ルール上 `origin='import'` に `user_accounts` が付くことは想定していないが、事故時の安全弁として毎回チェックする）。
2. **クリア**: `DELETE FROM parties WHERE origin='import' AND NOT EXISTS(user_accounts の紐付け)`。`party_roles` / `household_memberships` は `ON DELETE CASCADE` により自動的に削除される。
3. **parties 生成**: `household_members` の全行から `parties` を 1 行ずつ生成する。

   | parties カラム | 生成元 |
   | --- | --- |
   | `party_type` | 固定値 `'person'` |
   | `display_name` | `household_members.name` |
   | `display_name_kana` | `household_members.name_kana` |
   | `origin` | 固定値 `'import'` |
   | `email` / `phone` / `postal_code` / `address_1` / `address_2` | すべて `NULL`（連絡先の正は `households`。[パーティモデル設計.md](./パーティモデル設計.md) 「4.3」）|

4. **household_memberships 生成**: 生成した `party` ごとに 1 行生成する。

   | household_memberships カラム | 生成元 |
   | --- | --- |
   | `party_id` | 3. で生成した `party.id` |
   | `household_id` | `household_members.household_id` |
   | `member_role` | `household_members.member_role`（同値のまま） |
   | `succession_order` | `household_members.succession_order`（同値のまま） |
   | `household_member_id` | 生成元 `household_members.id`（生成元トレース） |

5. **party_roles 生成**: 3. で生成した `party` ごとに 1 行生成する（`role_type_id` = `role_types.code='parishioner'` の ID、`valid_from`/`valid_to` は NULL、`status='active'`）。`household_members` は全件「檀信徒」を表すため、ロールは一律 `parishioner` とする。
6. **report 記録**: `partiesImported` / `householdMemberships` / `partyRoles` の件数を実行ごとにレポートへ記録する（既存の件数サマリ・警告リストと同じ `report/migration-report.md` に追記する想定）。

### 7.2 origin ルールと冪等性

- `origin='import'` の `parties` は **`household_members` からの完全な派生データ**であり、フェーズ③実行のたびに **全削除 → 全再生成** する（差分更新はしない）。何度実行しても `household_members` が変わらなければ結果は同一になる（既存フェーズ①②の冪等性ポリシーと同じ考え方）。
- `origin='manual'`（寺務員・出店者・借主等、手入力）の `parties` および、それに付随する `party_roles` / `household_memberships` / `user_accounts` はフェーズ③のクリア（7.1 手順2）の対象外であり、**移行スクリプトからは一切触れない**。

### 7.3 CSV 洗い替え（フェーズ②）との関係

- フェーズ②は `PRODUCTION_TABLES`（`scripts/migrate/src/config.ts`）を `TRUNCATE ... RESTART IDENTITY CASCADE` する洗い替え方式（[README](../../scripts/migrate/README.md) 参照）。パーティモデル導入に伴い、**`household_memberships` を `PRODUCTION_TABLES` の先頭（最も子側）に追加する**。`household_memberships` は `households`/`household_members` への FK（`ON DELETE CASCADE`）を持つため、フェーズ②の `TRUNCATE` 対象に含めないと、②の洗い替え時に③で生成した行が親削除により暗黙 CASCADE 削除され、テーブル間の削除順序に暗黙依存する状態になってしまう。`TRUNCATE` 対象へ明示的に加えることで、②単体の再実行時にも `household_memberships` が確実に空になる。
- 一方で **`parties` / `party_roles` / `user_accounts` / `role_types` は `PRODUCTION_TABLES` に追加しない**。理由:
  - `parties` / `party_roles` は `origin='manual'` の不可侵データを含むため、②の `TRUNCATE`（CSV 起点の全削除）で巻き込むと手入力データが失われる。
  - `user_accounts` は認証情報であり、CSV 移行のたびに失効させてよいものではない。
  - `role_types` はテンプレ設定用マスタであり、CSV 移行の都度消してよいデータではない。
  - `origin='import'` 行の削除・再生成は、③自身の「全削除 → 全再生成」（7.2）が担う。②の `TRUNCATE` に含めなくても、③を実行すれば `household_members` 起点の派生データは常に最新化される。

### 7.4 実行モード

- `scripts/migrate/src/cli.ts` に **`--parties` 単独実行モード**を追加する。CSV 引数（`--meibo`/`--kakocho`）は不要で、DB に接続し「7.1 生成規則」のみを実行する（`--import`/`--transform` と同様、既存 DB の状態に対して追加実行できる独立モード）。
- 通常運用（`--all` / `npm run migrate`）では、**フェーズ②と同一トランザクション内で `transformDeceased`（`transform-deceased.ts`）の後にフェーズ③を実行する**。①②③を 1 トランザクションにまとめることで、途中失敗時にも `import_records`・本番 6 テーブル・`household_memberships` が中途半端な状態で残らない。

### 7.5 突合検証クエリ

移行後、`household_members` とパーティ生成結果が過不足なく対応していることを 3 種類のクエリで確認する。

**① 件数 3 点突合**（`household_members` 行数 = `parties(origin='import')` 行数 = `household_memberships` 行数 であること。`party_roles(parishioner)` も同数であること）

```sql
SELECT
  (SELECT COUNT(*) FROM household_members)                                  AS household_members_count,
  (SELECT COUNT(*) FROM parties WHERE origin = 'import')                    AS parties_import_count,
  (SELECT COUNT(*) FROM household_memberships)                              AS household_memberships_count,
  (SELECT COUNT(*) FROM party_roles pr
     JOIN role_types rt ON rt.id = pr.role_type_id
    WHERE rt.code = 'parishioner')                                          AS party_roles_parishioner_count;
-- 4 列すべてが一致していること（不一致は 7.1 手順1 のガードで除外された行がある可能性）
```

**② 属性突合**（`household_members` と、対応する `parties`/`household_memberships` の値が食い違っていないか）

```sql
SELECT hm.id AS household_member_id, hm.name, p.display_name,
       hm.name_kana, p.display_name_kana,
       hm.member_role, hmb.member_role AS membership_role,
       hm.succession_order, hmb.succession_order AS membership_succession_order
FROM household_members hm
JOIN household_memberships hmb ON hmb.household_member_id = hm.id
JOIN parties p ON p.id = hmb.party_id
WHERE hm.name IS DISTINCT FROM p.display_name
   OR hm.name_kana IS DISTINCT FROM p.display_name_kana
   OR hm.member_role IS DISTINCT FROM hmb.member_role
   OR hm.succession_order IS DISTINCT FROM hmb.succession_order;
-- 0 件であること
```

**③ 現戸主パリティ**（`household_memberships` の現戸主 1 件/世帯が、`household_members` の現戸主と同一世帯・同一人物を指しているか）

```sql
SELECT h.id AS household_id,
       hm.id  AS household_members_head_id,
       hmb.id AS household_memberships_head_id
FROM households h
LEFT JOIN household_members hm
       ON hm.household_id = h.id AND hm.member_role = 'head'
LEFT JOIN household_memberships hmb
       ON hmb.household_id = h.id AND hmb.member_role = 'head'
LEFT JOIN parties p ON p.id = hmb.party_id
WHERE (hm.id IS NULL) <> (hmb.id IS NULL)              -- 片方だけ現戸主が存在
   OR (hm.id IS NOT NULL AND p.display_name IS DISTINCT FROM hm.name); -- 現戸主の氏名不一致
-- 0 件であること（households.id ごとに head は高々 1 件という制約は両テーブルの部分 UNIQUE インデックスが担保する）
```

いずれも `report/migration-report.md`（[README](../../scripts/migrate/README.md) 「レポートの読み方」）の件数サマリに準じ、③実行後のレポートへ突合結果を追記する運用を想定する。
