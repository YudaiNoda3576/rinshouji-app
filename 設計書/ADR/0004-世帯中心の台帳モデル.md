# 0004: households を中核に構成員・故人・墓地・納骨堂を従属させる世帯中心の台帳モデルとする

- ステータス: 承認
- 決定日: 2026-07-14
- 決定者: DB設計担当

## コンテキスト

移行元の檀信徒名簿 CSV は「家番」を単位とした戸主中心の世帯台帳であり、1行が1世帯（家）に対応する。過去帳 CSV も多くの行が家番を介して世帯に紐づく。この実データの構造に対して、正規化モデルの中心エンティティを何に置くかを決める必要があった。

現行 UI（`ParishFamily`）も家名（世帯）を主表示・検索キーとしており、既存アプリとの整合も要件になる。

## 検討した選択肢

| 観点 | 世帯中心モデル（採用） | 個人（person）中心モデル |
| --- | --- | --- |
| 実データとの対応 | 名簿1行=1世帯に直結し、移行が単純 | 名簿から個人を抽出する追加の名寄せ・分解ロジックが必要 |
| 行事設定（般若・施食・棚経等）の置き場 | 世帯に対して単一値のため households に自然に内包できる | 世帯という概念が弱まり、行事設定の置き場が個人・別テーブルのいずれか曖昧になる |
| 家名検索・現行UIとの整合 | `households.family_name` がそのまま檀家一覧の主表示・検索キーになる | 個人の姓を家名として扱うか別概念にするか、追加の設計判断が必要 |
| 孤立故人（家番8888等）の表現 | `deceased_persons.household_id` を NULL 可にするだけで表現できる | 個人中心では「世帯不明の個人」という例外が本流に混ざりやすい |

## 決定

`households`（世帯）を中核テーブルとし、`household_members`（構成員）・`deceased_persons`（故人）・`cemetery_plots`（墓地区画）・`cemetery_plots`・`columbarium_units`（納骨堂区画）を世帯に従属させる構成とする。

- 現戸主は `household_members` に `member_role='head'` として持ち、世帯あたり1件を部分 UNIQUE インデックス（`UNIQUE(household_id) WHERE member_role='head'`）で担保する。
- 故人は世帯に紐づく場合と紐づかない場合があり、`deceased_persons.household_id` を NULL 可とする。過去帳の家番 8888（意味不明、57件）や「削除」「抹消」「空」の行は、対応世帯を特定できないため FK を NULL とし、孤立故人として扱う（表示用に `legacy_house_no`/`legacy_district_1`/`legacy_district_2` を保持。[0001](./0001-csv原文のimport-recordsステージング.md) 参照）。

## 理由

- 移行元データの構造（家番中心の名簿）に最も自然に対応し、CSV→本番テーブルの変換ロジックを単純化できる。
- 般若・施食・棚経・月参り・地蔵といった行事設定は世帯に対して単一値であり、世帯中心モデルであれば `households` に内包するだけで表現でき、別テーブル化による無駄な JOIN が発生しない。
- 現行 UI（`ParishFamily.name`）や檀家一覧の主表示・検索キーとの整合が取れる。

## 結果・トレードオフ

- 世帯に属さない個人（孤立故人）を正規に表現するために `household_id` を NULL 許容にしており、NULL 分岐を考慮した画面ロジック・クエリが必要になる。
- 将来「檀信徒個人を中心に据えた機能」（例: 個人ごとのポータルログイン）が必要になった場合は、世帯中心モデルの上に個人主体の概念を別途重ねる必要がある。これは [パーティモデル設計.md](../DB設計/パーティモデル設計.md) の `parties`（[0006](./0006-パーティモデルの採用.md)）が担う。
