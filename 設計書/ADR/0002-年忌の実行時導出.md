# 0002: 年忌テーブルを持たず death_year から実行時に12マイルストーンを導出する

- ステータス: 承認
- 決定日: 2026-07-14
- 決定者: DB設計担当

## コンテキスト

過去帳 CSV には「年忌」列（一周忌・三回忌…）が存在するが、この値は「令和8年時点」という特定年のスナップショットにすぎない。年忌（一周忌・三回忌・七回忌…）は没年月日と基準年（実施年 = 没年 + 経過年数）から機械的に算出できる値であり、時間の経過とともに古い年忌ラベルは意味を失っていく。

檀家一覧・過去帳一覧に加えて「年忌案内一覧」画面が必要であり、年忌をどのようにデータとして持つか（保持するか、都度算出するか）を決める必要があった。

## 検討した選択肢

| 案 | 内容 | 評価 |
| --- | --- | --- |
| 年忌テーブル事前生成 | 故人ごとに12マイルストーン分の行を事前生成するテーブル（例 `memorial_anniversaries`）を持ち、年ごとのバッチ等で更新する | 基準年が変わるたびに更新処理が必要になり、更新漏れによる不整合リスクがある。故人1名につき12行を生成・管理するコストも発生する |
| **実行時導出（採用）** | 年忌テーブルを持たず、`deceased_persons.death_year` と12種のマイルストーン定義（`n` = 没年からの経過年数）から、参照時に `実施年 = death_year + n` を機械算出する | 更新処理・同期が一切不要で、常に最新の基準年に対して正しい年忌が算出される。ストレージ・更新コストもゼロ |

## 決定

年忌を保持する専用テーブルは持たない。`deceased_persons.death_year`/`death_month`/`death_day` のみを保持し、年忌案内は参照時（バックエンド API・フロントエンド双方）にマイルストーン定義から機械導出する。

- マイルストーン定義（12種）は `app/backend/src/routes/notices.ts` の `MILESTONES` と `app/frontend/src/features/memorial/utils.ts` の `ANNIVERSARY_MILESTONES` の2箇所に、同一内容を意図的に複製して持つ（コメントで明記）。
- 過去帳 CSV 原文の「年忌」列（令和8年時点のスナップショット値）は本番テーブルに持たず `import_records.raw_data` にのみ残す（[0001](./0001-csv原文のimport-recordsステージング.md) の判定基準に合致）。

## 理由

- 年忌は没年月日から一意に決まる導出値であり、別テーブルに事前生成すると「更新同期の手間」と「基準年経過による不整合リスク」を抱え込むだけで正規化上の利得がない。
- 実行時導出であれば、いつ参照しても常に現在時点で正しい年忌一覧が得られ、バッチ更新・同期処理が構造的に不要になる。

## 結果・トレードオフ

- マイルストーン定義をバックエンド・フロントエンドの2箇所に複製しているため、マイルストーンの追加・変更時は両方の同時更新が必要（コメントで明記し、レビューで気づけるようにする運用でカバー）。
- 正式な法要実施記録（実際に法要を行ったかどうか）を管理する要件が将来出た場合は、導出値とは別に実施記録テーブルを追加する必要がある（本 ADR の対象外）。
- 不完全な没日付（`death_month`/`death_day` が NULL）の故人は、年月日が揃わない範囲で導出可能な情報のみが年忌一覧に反映される。
