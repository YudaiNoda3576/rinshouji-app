# 0003: ORM を使わず生 SQL + 01_schema.sql 一本でスキーマを管理する

- ステータス: 承認
- 決定日: 2026-07-17
- 決定者: DB設計担当

## コンテキスト

PostgreSQL 上に正規化モデルを構築するにあたり、スキーマ定義とマイグレーションをどう管理するか（ORM/マイグレーションツールを導入するか、生 SQL で直接管理するか）を決める必要があった。

本プロジェクトは初回移行（Excel 由来 CSV → PostgreSQL）が主目的であり、`postgres:16-alpine` コンテナ初回起動時に `/docker-entrypoint-initdb.d` 配下の SQL が自動実行される構成を前提としている。バックエンドは Node.js + `pg` ドライバのみを採用している。

## 検討した選択肢

| 案 | 内容 | 評価 |
| --- | --- | --- |
| ORM/マイグレーションツール導入（Prisma・TypeORM・Drizzle 等） | スキーマを DSL で定義し、増分マイグレーションファイルを自動生成・自動適用する | 型安全性やマイグレーション管理の恩恵がある一方、導入・学習コストが発生し、生 SQL で書ける複雑な CHECK 制約・部分 UNIQUE インデックス・自己参照 FK 等を DSL で表現しづらい場面がある |
| **生 SQL + 01_schema.sql 一本管理（採用）** | `pg` ドライバのみを使用し、スキーマ定義を `db/init/01_schema.sql` 1ファイルに集約する。冪等性は先頭で `DROP TABLE IF EXISTS ... CASCADE`（依存関係の逆順）を行うことで担保する | 複雑な制約（部分 UNIQUE、`NULLS NOT DISTINCT`、CHECK 制約の組み合わせ）を SQL で直接・正確に表現できる。ツール学習コストがなく、DDL がそのまま設計の正になる |

## 決定

ORM やマイグレーションツールは導入しない。`db/init/01_schema.sql` を唯一のスキーマ定義ファイルとし、`DROP TABLE IF EXISTS ... CASCADE`（子→親の順）で冪等に再構築できる形で管理する。バックエンドは `pg` を用いた生 SQL クエリで実装する。

## 理由

- 初回移行プロジェクトという性質上、まず正確な正規化スキーマを1回で構築することが最優先であり、増分マイグレーションの automation よりも DDL の表現力・可読性を優先した。
- 部分 UNIQUE インデックス（現戸主1件/世帯）、`NULLS NOT DISTINCT`、FK ごとに異なる `ON DELETE` 方針など、ORM の DSL では素直に表現しづらい制約が多い。
- スキーマの正が SQL ファイル1本に集約されるため、レビュー・差分確認が容易。

## 結果・トレードオフ

- **稼働後の増分マイグレーションは手動運用になる。** `01_schema.sql` は `DROP → CREATE` の冪等スクリプトであり、既存データを保持したままのスキーマ変更（列追加等）を自動生成する仕組みがない。運用開始後にスキーマ変更が必要になった場合は、別途手動での `ALTER TABLE` スクリプトを都度作成する必要があり、変更履歴の一元管理ができていない点はリスクとして明記する。
- 型安全性（TypeScript の型とスキーマの自動同期）は得られず、クエリの型はアプリ側で手動定義する。
- マイグレーションツールが提供するロールバック機能がないため、本番適用前の検証（ステージング環境での事前適用等）がより重要になる。
