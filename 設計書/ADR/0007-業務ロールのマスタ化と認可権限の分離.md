# 0007: 業務ロールはマスタテーブル化し、認可権限（permission）はシステム enum のまま分離する

- ステータス: 承認
- 決定日: 2026-07-19
- 決定者: DB設計担当

## コンテキスト

パーティモデル（[0006](./0006-パーティモデルの採用.md)）導入にあたり、業務ロール（寺務員・檀信徒・出店者・利用者…）をどう表現するかを決める必要があった。本システムは林昌寺専用に閉じたものではなく、将来的に他寺院へテンプレートとして展開し、寺院ごとに個別最適化する運用を想定している（[0009](./0009-単一寺院前提とテンプレ展開方針.md)）。この前提が、業務ロールをスキーマにハードコードすべきかデータとして持つべきかの判断に強く影響する。

あわせて、`user_accounts.permission`（admin/staff/readonly）という既存の認可の仕組みと、新設する業務ロールをどう関係づけるかも決める必要があった。

## 検討した選択肢

### ロールの表現方法

| 観点 | `role_types` マスタ化（採用） | CHECK 制約案 |
| --- | --- | --- |
| 他寺院テンプレ展開時の追加コスト | `INSERT` のみ。無停止で追加可能 | `ALTER TABLE ... ADD CONSTRAINT` が必要。リリースを伴う |
| 既存データへの影響 | 影響なし | 制約変更時に既存値との整合性チェックが必要 |
| 表示順・説明文の管理 | `sort_order`/`label`/`description` 列で一元管理 | アプリ側コードにハードコードが必要 |
| ロールの無効化 | `is_active=false` で論理無効化。履歴（`party_roles` の過去行）は保持されたまま表示から除外できる | 値を削除できない（既存行が参照している値を CHECK から外せない） |

### 業務ロールと認可権限の関係

業務ロール（`party_roles`/`role_types`＝「その人が寺との関係で何者か」）と、認可権限（`user_accounts.permission`＝「システムに何をしてよいか」）を同一概念として扱うか、直交する別概念として分離するかを検討した。同一概念として扱う案（例: ロールから権限を自動導出する）は、「`staff` ロールを持つが `readonly` 権限の職員」のような組み合わせを表現できず、[0006](./0006-パーティモデルの採用.md) が想定する将来の檀信徒ポータル（`parishioner` ロール + `readonly` 権限）にも対応できないため、直交する別概念として分離する案を採用した。

## 決定

- `role_types` は業務ロールを保持するマスタテーブルとし、`party_roles.role_type_id` から FK 参照する。テンプレ展開時はデータ（`INSERT`/`is_active` 更新）の差し替えのみで対応する。
- `party_type`（person/organization）・`parties.status`/`party_roles.status`・`user_accounts.permission`/`status` は、コードが if/switch で直接分岐に使う・寺院間で変わらないシステム enum として、従来どおり VARCHAR + CHECK 制約のまま維持する。
- 業務ロール（`party_roles`）と認可権限（`user_accounts.permission`）は直交する別概念として分離する。業務ロールは 0..N（同一パーティが複数ロールを兼ねうる）、認可権限はアカウントごとに1つ（CHECK 列）。

**使い分け基準**: 寺院ごとに増減・カスタマイズされうる「業務データ」的な値（`role_types`）はマスタテーブル化し、コードが直接分岐に使う・寺院間で変わらない「システム enum」的な値（`party_type`/`status`/`permission`）は CHECK 制約のままとする。この基準は既存の `districts`（地区マスタ）や `import_records.source_file`（CHECK のまま。移行元ファイル種別はシステム固有で変わらない）とも整合する。

## 理由

- 業務ロールは寺院ごとの運用に左右されるデータであり、CHECK 制約にハードコードすると寺院間の役割の違いに対応するたびマイグレーションが発生する。マスタ化すれば `INSERT` だけで済む。
- 業務ロールと認可権限を分離することで、「業務上は檀信徒だが閲覧のみ許可するポータルアカウント」のような組み合わせをスキーマ変更なしに表現できる。
- `permission` はフロント（`SectionStaff.tsx` の admin/staff/readonly 判定）と直接対応するシステム enum のため、role_types 化の対象外とする。

## 結果・トレードオフ

- ロール種別への参照が FK（`role_type_id`）になるため、CHECK 制約に比べてクエリ時に JOIN が1段増える。
- `role_types` の初期シード（staff/parishioner/vendor/tenant）はテンプレ既定値であり、他寺院展開時にどこまで差し替えてよいか（コード自体は不変・ラベルや sort_order のみ変更可能等）の運用ルールは本 ADR の範囲外で別途定める必要がある。
- 業務ロールと認可権限を分離した結果、両者の整合性（例: `staff` ロールを持たないパーティに `staff` 権限相当のアカウントが付与される等）はアプリ側のバリデーションに委ねられ、DB 制約では防げない。
