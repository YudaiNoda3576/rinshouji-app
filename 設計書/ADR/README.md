# ADR（Architecture Decision Record）

ADR は、システムの設計・実装において行った重要な意思決定を「なぜそう決めたか」という理由とともに記録するドキュメントである。検討した選択肢・決定・トレードオフを残すことで、後から設計を見た人（将来の開発者・自分自身）が背景を追跡でき、状況が変わった際に「なぜこうなっているか」を踏まえて再検討できるようにする。

## 運用ルール

- **ADR は不変**である。一度承認した ADR の本文（コンテキスト・決定・理由等）は編集しない。
- 決定を変更する場合は、既存 ADR を書き換えるのではなく **新しい ADR を起こす**。旧 ADR は本文を変更せず、**ステータス行のみ**を「置換済み(→ 新No)」に更新する（変更してよいのはステータス行のみ）。
- バージョン管理の実体は git 履歴と、上記の置換チェーン（旧 ADR のステータス行が新 ADR を指す）である。

## 役割分担（ADR と設計書）

- **ADR**: 「なぜそう決めたか」という意思決定の記録。検討した選択肢・採用理由・トレードオフに焦点を当てる。
- **設計書**（[DB設計/](../DB設計/) 等）: 「何をどう作るか」の記述。ER図・カラム定義・移行規則等の詳細な仕様そのもの。

内容を二重管理しないため、ADR には仕様の詳細を書き写さず、詳細は該当する設計書へのリンクで誘導する。

## 命名規約

`NNNN-<短い日本語タイトル>.md`（連番4桁、ハイフン区切り）。新規作成時は [template.md](./template.md) を複製して使う。
